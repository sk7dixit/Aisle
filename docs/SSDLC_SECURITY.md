# Secure Software Development Lifecycle (SSDLC) & DevSecOps Checklist

This guide documents mandatory security policies, code review checklists, and deployment rules for the Aisle engineering team. Developers must adhere to these policies for all pull requests and releases.

---

## 1. Branch Protection & GitHub Security Rules

### Repository Settings (`main` branch)
* **No Direct Push**: All changes must be proposed via a branch and merged via Pull Request.
* **Review Required**: Minimum of 1 peer security review approval is required before merging.
* **Signed Commits**: Enforce commit signature verification (GPG/SSH keys) to verify the identities of all committers.
* **Checks Required**: The CI/CD pipeline (tests, SAST scanner, dependency audit) must pass before a pull request can be merged.

### Secrets Management
* **Never commit secrets**: No `.env`, private keys, API secrets, database credentials, or passwords may be stored in code.
* **Production Configurations**: Use GitHub Secrets for CI/CD and secure runtime secrets managers (AWS Secrets Manager / HashiCorp Vault) for live environment variables.
* **Secret Scanning**: Gitleaks or TruffleHog must be installed locally and as a pre-commit hook to catch secrets before they are pushed:
  ```bash
  gitleaks detect --verbose
  ```

---

## 2. Dependency & Supply Chain Security

* **Version Pinning**: All dependencies in `package.json` must be pinned to exact versions (e.g., `"express": "5.2.1"` instead of `"^5.2.1"`) to prevent unpredictable upstream packages from introducing vulnerabilities during builds.
* **Weekly Audits**: Run `npm audit` locally weekly.
* **CI Build Policy**: The build pipeline runs `npm audit --audit-level=critical`. Any critical vulnerability discovered in dependencies will fail the build.

## 3. AI-Generated Code Security Checklist

AI assistants and LLM agents generate a significant volume of application code. Developers must verify that AI-generated snippets comply with the following checklist before submitting a pull request:

| Security Domain | Verification Standard | Check |
| :--- | :--- | :---: |
| **Authentication** | Ensure route handlers are protected by the `protect` middleware where required. | [ ] |
| **Authorization** | Verify that administrative actions are locked behind the `authorize('super_admin', 'admin')` guard. | [ ] |
| **Input Validation** | Confirm that request parameters are validated using Joi schemas (`validateBody`) before controller execution. | [ ] |
| **NoSQL Injection** | Never pass un-sanitized user payloads directly to database query filters (e.g. avoid `User.find(req.body)`). | [ ] |
| **Command Injection** | Avoid `child_process.exec()` or `eval()` with user inputs. | [ ] |
| **Audit Logs** | Check that sensitive actions (creates, deletes, role updates) write audit trails to `logSecurityEvent`. | [ ] |
| **Secrets Exposure** | Verify that no temporary API keys or mock passwords generated by the AI are left in the code. | [ ] |

## 4. Secure Development Lifecycle (SSDLC) Flow

```
1. Planning & Design ──> 2. Threat Modeling ──> 3. Secure Development
                                                          │
6. Prod Monitoring <── 5. Secure Release <── 4. CI/CD & Testing
```

1. **Planning & Design**: Define features alongside security requirements (e.g., data encryption, rate limit boundaries).
2. **Threat Modeling**: Map out system inputs, data flows, and potential exploit vectors before writing complex code.
3. **Secure Development**: Write code aligning with secure coding standards and run the local static analyzer (`node backend/scripts/securityScan.js`).
4. **CI/CD & Testing**: Automatic execution of dependency checks, secret scans, SAST, and integration test suites.
5. **Secure Release**: Administrative changes, payment handlers, database updates, and role modifications require explicit approval from the lead engineer.
6. **Production Monitoring**: Real-time logging of anomalies in the Security Operations Center (SOC) dashboard.
